# SysFolio TOC Activation Strategy

## 文档目的

这份文档定义目录高亮（TOC active state）的视觉与交互规则，用来解决以下问题：

- 正文已经滚动到底
- 用户主观上认为“已经读到最后”
- 但 TOC 仍然停留在倒数第二项

当前假设：

- 暂时没有 footer
- 暂时没有评论区
- 暂时没有推荐内容
- TOC 只映射正文主内容结构

在这个前提下，“正文滚动到底”应当被视为“最后一节已到达”。

## 设计目标

1. TOC 高亮应尽量反映“当前阅读章节”。
2. 用户滚到底时，不应出现“目录还没到最后”的违和感。
3. 中段滚动切换应自然，不能太晚。
4. 规则应适用于长章节、短章节和最后一节较短的情况。

## 结论

推荐采用组合方案：

`激活线 + 到底兜底 + 正文底部留白`

这三者同时成立时，体验最稳定。

## 一、主规则：激活线

TOC 当前项不建议按“标题碰到视口顶边”来判断，而建议按“标题进入激活线”来判断。

建议规则：

- 在正文滚动容器内设置一条激活线
- 激活线位置建议在视口顶部下方约 `25%~33%`
- 当某个标题进入或越过这条线时，将对应 TOC 项设为当前项

设计原因：

- 比顶边判断更符合人的阅读感知
- 章节切换不会太晚
- 对长文阅读更自然

## 二、到底兜底

当正文容器已经滚动到底时，直接将 TOC 激活到最后一项。

建议规则：

- 当 `scrollTop + clientHeight >= scrollHeight - epsilon`
- 直接激活最后一个 TOC 项

其中：

- `epsilon` 只作为误差容忍，避免浮点或像素舍入导致临界抖动
- 判断对象应是正文滚动容器，而不是整个页面

设计原因：

- 解决“已经读完，但 TOC 还没到最后”的违和感
- 这是用户主观预期最强的场景，应该优先顺从预期

## 三、正文底部留白

正文末尾应保留一段底部留白，让最后一个章节在视觉上有机会真正进入阅读区，而不是贴着页面底边结束。

建议方向：

- 在正文内容区增加适度 bottom padding
- 目的不是制造空白，而是给最后一节留下进入激活线的空间

设计原因：

- 可以减少最后一节过短时的判定问题
- 让阅读结束时的视觉节奏更从容
- 能降低对“到底兜底”的依赖

## 四、为什么不用其他方案

### 1. 不推荐只用“标题到顶才激活”

问题：

- 最后一节如果很短，经常永远激活不到
- 章节切换整体偏晚

### 2. 不推荐只用“滚动百分比”

问题：

- 百分比表达的是阅读进度，不是当前章节
- 可能出现章节很短但高亮跳转不合理的情况

### 3. 不推荐只用“到底时切最后一个”

问题：

- 中间过程没有改善
- 仍然会在接近底部时显得切换太晚

## 五、适用于当前 SysFolio 的规则

在当前没有 footer、评论区、推荐内容的前提下，TOC 高亮建议按以下优先级工作：

1. 如果正文容器已到底，激活最后一个 TOC 项
2. 否则按激活线判断当前章节
3. 正文末尾保留额外底部留白，帮助最后一节自然进入激活区

## 六、交互解释

这套规则表达的是：

- TOC 代表“当前正文章节”
- 不是“整个页面滚动位置”
- 也不是“纯阅读进度条”

因此，当正文已经结束时，TOC 应与“用户已到最后一节”的心理模型保持一致。

## 七、给前端的实现约束

1. 以正文滚动容器为判断基准，不以整页文档高度为准。
2. 激活线优先，不要只看标题是否触顶。
3. 到底判断必须保留，作为强兜底。
4. 底部留白应放在正文内容区，不建议用页面外层 hack。
5. 若后续加入 footer、评论区或推荐内容，需要重新定义“TOC 映射正文结束”还是“映射整页结束”。

## 八、当前设计判断

对于当前 SysFolio，这不是一个“视觉特效”问题，而是阅读定位规则问题。  
最合适的处理方式是：

`激活线 + 到底兜底 + 正文底部留白`

这也是当前最推荐给前端实现的方案。
