# SysFolio 文件系统式阅读页 0.5 期基线设计

## 1. 概述

### 1.1 版本定位

这份文档定义 `0.5 期` 的阅读页基线能力：

- 目标是先把文件系统式阅读框架立住。
- 重点解决三栏骨架、阅读体验、轻量树搜索、夜间模式与阅读进度。
- 不包含 1 期的“树内统一搜索与高级筛选”增强方案。

1 期方案请见 `filesystem-reading-page-v1.md`。

### 1.2 项目定位

SysFolio 不是一个传统博客首页，也不是纯文档站。它的核心心智是“个人数字档案 / 作品展柜”：

- 用户像在浏览作者的本地文件系统。
- 文件夹承载主题、系列、项目或时间线。
- 文件承载具体内容，包括文章、小游戏、多媒体作品等。
- 0.5 期只支持文本内容打开与阅读，但整体结构必须为未来的非文本类型预留空间。

### 1.3 0.5 期设计目标

本期设计聚焦五件事：

1. 让用户在首屏内理解“左边找内容，中间看当前目录或当前文件”的页面逻辑。
2. 用极简、可关闭的新手引导降低第一次进入时的理解门槛。
3. 让文本文章的阅读体验足够稳定、舒适、持续。
4. 让目录在内容增多时仍然具备定位效率，而不是沦为噪音树。
5. 在不打断阅读的前提下，明确表达这个系统未来不只有文章。

### 1.4 非目标

0.5 期不做以下能力：

- 不做真实文件管理行为，例如拖拽、重命名、排序、批量操作。
- 不做小游戏、多媒体内容的实际打开与播放。
- 不做拟真的桌面操作系统复刻，不引入复杂窗口控制和强装饰性假交互。
- 不以“像 IDE”作为首要目标，阅读质量高于拟真程度。
- 不做首次点击文件夹或文件后的二次教学提示，避免持续打扰。

### 1.5 配套文档

本设计说明由以下配套文档补充落地细节：

- `visual-tokens.md`：定义颜色、排版、间距、圆角、描边、阴影和动效 token。
- `component-style-guide.md`：定义通用组件结构、尺寸、状态和响应式规则。

## 2. 用户心智与核心任务

### 2.1 用户理解

用户进入页面后，应该在几秒内形成以下理解：

- 这是作者的个人数字空间，不是普通博客列表。
- 左侧是目录树，可以浏览不同主题与文件。
- 当前选中的目录或文件会在中间显示对应内容。
- 现在主要能读文章，未来还会有其他类型的内容。

### 2.2 核心任务

0.5 期的主任务路径为：

1. 进入页面。
2. 通过轻量引导理解路径栏、文件树和内容区的职责。
3. 浏览或搜索左侧文件树，找到感兴趣的文件。
4. 打开文本文件并进入阅读。
5. 在阅读中通过右侧目录或上下文信息继续定位。
6. 读完后跳转到同目录或同系列的下一个内容。

### 2.3 主要矛盾

这个页面最容易失败的点不是配色，而是结构冲突：

- 如果文件系统感太强，用户会一直浏览而不进入阅读。
- 如果阅读器感太强，页面又会退化成普通博客详情页。

因此设计边界必须明确：

- 文件系统隐喻只服务导航、定位和内容扩展预期。
- 正文区域必须优先服从长文阅读规律。

## 3. 页面结构

### 3.1 整体骨架

桌面端采用三栏布局：

- 左栏：`FileTree`
- 中栏：`ContentPane`
- 右栏：`ContextPanel`

并在顶部加入一条轻量路径栏：

- 顶部：`PathBar`

`ContentPane` 固定支持三种主视图状态：

- `HomeView`
- `DirectoryView`
- `DocumentView`

推荐信息权重：

- 中栏最大，承担当前节点的主内容；首页节点显示首页态，目录节点显示目录态，文章节点显示文件态。
- 左栏偏窄，只负责浏览与定位。
- 右栏偏轻，只提供当前节点的上下文信息。
- 新手引导层只在首次进入时短暂出现，不改变三栏主结构。

### 3.2 区域职责

#### PathBar

作用：

- 告诉用户当前所在路径。
- 作为全局面包屑，表达当前节点的真实层级。
- 强化“当前已选中节点”的状态，而不局限于文件。
- 提供最轻量的返回上一级语义。
- 承载夜间模式切换等少量全局工具。

约束：

- 不做重型导航栏，不放过多全局入口。
- 不使用明显的浏览器式 UI 复刻。
- 信息顺序采用“根目录 / 文件夹 / 当前节点”。
- 面包屑逐级可点击，但当前节点不表现成主按钮。
- 右侧工具数量严格收敛，默认只容纳主题切换与少量帮助入口。

#### FileTree

作用：

- 承担文件系统隐喻。
- 展示内容分组关系与未来的文件类型扩展性。
- 作为页面主导航。
- 通过搜索与层级控制维持定位效率。

约束：

- 层级不可过深，首屏默认最多展示 2 到 3 层结构。
- 节点命名应像内容目录，而不是技术目录。
- 默认不显示过多尺寸、时间戳、权限等噪音字段。
- 顶部固定轻量搜索框，支持按文件夹名和文件名搜索。
- 默认展开 2 层目录，第 3 层及更深层级默认折叠。
- 深层目录按需加载，避免首屏为完整树付出加载成本。

#### ContentPane

作用：

- 作为中间主内容区，承载当前选中节点的主要内容。
- 当节点是首页节点时进入 `HomeView`，展示个性化首页内容与精选入口。
- 当节点是目录时进入 `DirectoryView`，优先展示当前目录下的子目录与子文件。
- 当节点是文章时进入 `DocumentView`，进入标准阅读态。
- 为未来 `game`、`media` 等内容类型的占位态预留统一承载容器。

约束：

- 不做第二棵文件树，层级浏览仍以左栏和面包屑为主。
- `HomeView` 允许少量定制内容块，但不扩张成第二套站点首页系统。
- `DirectoryView` 只展示当前目录的一层子项，不递归展开深层树。
- 打开文章后，注意力必须自然落到正文标题和前两屏内容。
- 不让左右两栏的 hover、动画、色块打断阅读。
- 在恢复阅读进度时提供轻量反馈，而不是强制用户重新定位。

#### ContextPanel

作用：

- 展示当前节点的上下文信息。
- 在首页态下提供首页说明、精选入口或最近更新摘要。
- 在目录态下提供目录说明、子项统计与相关入口。
- 在文件态下提供目录、标签、日期、阅读时长、系列关系与文内目录。

约束：

- 不承载主导航。
- 不出现和左栏重复的文件树结构。
- 信息要围绕“当前节点”，不能做成另一个内容堆栈。

#### OnboardingHints

作用：

- 在首次访问时帮助用户快速理解三栏职责。
- 用最少的视觉介入说明 FileTree、ContentPane 和 PathBar 的含义。

约束：

- 仅首次进入页面自动出现一次。
- 页面默认显示 3 个半透明提示点，不直接展开长文案。
- 点击提示点后，才显示一句话说明；点击空白处关闭整个引导层。
- 首次点击文件夹或文件时不追加任何提示。

## 4. 内容模型与状态

### 4.1 文件类型

0.5 期先在设计层固定以下文件类型：

- `folder`
- `home`
- `article`
- `game`
- `media`
- `unknown`

### 4.2 文件状态

0.5 期先在设计层固定以下状态：

- `available`：当前可打开并完整展示。
- `coming_soon`：未来会支持，但当前不可用。
- `external`：未来可能跳转到站外或独立容器。

### 4.3 0.5 期行为定义

- `home + available`：进入 `HomeView`。
- `folder + available`：进入 `DirectoryView`。
- `article + available`：进入 `DocumentView`。
- `game + coming_soon`：进入统一空态，不报错。
- `media + coming_soon`：进入统一空态，不报错。
- `unknown`：显示保守占位，不打断整体浏览。

统一空态应表达两件事：

- 这是一个合法文件类型。
- 这个类型后续会支持，但当前版本仍以文本为主。

### 4.4 HomeNode 特例规则

- `home` 是轻量特例节点，不要求走复杂的统一内容建模。
- 只需要最小字段：节点名称、图标、目标视图、少量手写摘要或精选入口。
- 默认放在根层级或根层级下的固定分组中，不参与深层目录递归。
- 视觉上与普通目录节点保持同一体系，只允许轻量定制，不做强品牌化 landing page。

### 4.5 阅读态附加状态

0.5 期同时引入以下阅读态状态：

- `progress_saved`：已保存当前文章的滚动位置。
- `progress_restored`：重新进入文章时恢复到上次阅读位置。
- `theme_light` / `theme_dark`：内容区与面板区同步切换主题 token。

## 5. 视觉语言

### 5.1 总体方向

整体风格借鉴现有 VS Code `Day / Care` 与 `Night / Focus` 主题，但不直接复刻编辑器：

- 背景偏暖米色，带轻微纸面感。
- 面板使用低饱和分区色，形成稳定层次。
- 用少量冷色强调选中态和主交互。
- 用少量暖色提示定位与阅读进度。
- 错误色只在异常态出现，不参与日常层级。
- 夜间模式沿用同一套语义结构，只切换 token，不改组件骨架。

气质关键词：

- 档案
- 纸面
- 本地目录
- 作者工作区
- 克制而有结构

### 5.2 颜色角色

建议抽象为 6 组 token：

- `canvas`：正文背景，暖纸面色。
- `panel`：侧栏与路径栏背景，比正文略深。
- `text`：深褐灰正文色，避免纯黑。
- `accent`：低饱和蓝，用于选中、链接、聚焦。
- `signal-warm`：赭色，用于阅读进度、轻提示、章节定位。
- `danger`：柔和红，用于错误与无效态。

### 5.3 层级策略

- 最大对比留给标题、正文和当前选中项。
- 中对比留给路径栏、目录标题、标签和局部说明。
- 低对比留给面板背景、分割线、次要辅助信息。

避免的视觉问题：

- 侧栏与正文对比度过接近，导致结构塌陷。
- 所有信息都像“当前可操作”，形成噪音。
- 过多描边、阴影和拟物按钮，让界面发脏。

### 5.4 排版策略

正文必须明显不同于编辑器文本，并采用偏阅读友好的无衬线排版：

- 正文字体默认使用无衬线字体。
- 基础字号默认 `16px`。
- 行高默认 `1.8`。
- 行宽控制在 `60` 到 `80` 个字符之间，避免超宽阅读。
- 段落间距默认 `1.2em`。
- 引用、列表、代码块、脚注拥有独立层级。

路径、文件名和代码仍可使用等宽字体，但正文不走编辑器式密排。

## 6. 交互规则

### 6.1 首次引导

- 首次进入页面时，默认显示 3 个半透明提示点，分别锚定 `FileTree`、`ContentPane`、`PathBar`。
- 提示点默认不展开文案，保持页面主体可见。
- 点击提示点时，显示一句话说明：
  - `FileTree`："左侧浏览文件和目录。"
  - `ContentPane`："中间查看首页、目录或阅读当前文件。"
  - `PathBar`："这里查看当前所在路径并切换主题。"
- 点击页面空白处时，关闭整个引导层并记为已看过。
- 首次点击文件夹或文件时，不额外弹出说明。

### 6.2 文件树交互

- 首页仅加载前 2 层目录。
- 第 3 层及以上目录默认折叠。
- 点击文件夹展开或收起；若该文件夹尚未加载其子层级，则在展开时按需加载。
- 点击首页节点时，进入轻量 `HomeView`。
- 点击文章文件打开正文。
- 当前打开文件在树中保持持续高亮。
- 文件树 hover 反馈轻量，不做强色块追踪。
- 如果打开的是未支持类型，进入统一空态而不是弹错误。
- 当目录超过首屏高度时，左栏内部独立滚动，不推动正文区错位。

### 6.3 搜索与层级逻辑

- `FileTree` 顶部固定轻量搜索框。
- 搜索范围仅包含文件夹名和文件名，不搜索正文内容。
- 搜索结果高亮命中片段。
- 搜索状态下自动展开命中节点的祖先路径，方便直接定位结果。
- 清空搜索后恢复到默认展开策略，而不是保留整棵树完全展开。

### 6.4 路径与选中同步

以下三个位置的状态必须同步：

- `PathBar`
- `FileTree` 当前节点
- `ContentPane` 当前头部

同步目标是降低迷路风险。用户随时都能知道：

- 我现在选中的是哪个目录或文件。
- 它位于哪个路径层级。
- 我还能回到哪里。

### 6.5 内容区交互

- 选中首页节点时，中间区显示 `HomeView`，可容纳少量定制内容块与精选入口。
- 选中目录时，中间区显示 `DirectoryHeader`、`DirectoryMeta` 与 `DirectoryChildren`。
- `DirectoryView` 中子目录优先于文件展示，避免目录与文章混排失序。
- 从目录态点击子目录时，继续进入下一级 `DirectoryView`。
- 从目录态点击文章时，切换到 `DocumentView`。
- 目录态只展示当前目录的一层子项，不在中间区递归模拟树形结构。

### 6.6 阅读区交互

- 标题区显示标题、日期、标签、阅读时长。
- 文内标题自动同步右侧目录高亮。
- 点击右侧目录时平滑定位到对应章节。
- 切换文章时，中栏仅保留轻量过渡，用于表达“打开新文件”，不制造拖沓感。
- 正文默认支持原生复制，不对选择文本做干扰性装饰。
- 滚动阅读过程中自动保存当前文章阅读位置。
- 用户重新进入同一文章时，内容渲染完成后自动恢复到上次位置。
- 恢复后显示轻量提示，允许用户一键回到顶部。

### 6.7 文内元素规则

- 代码块提供语法高亮，语义颜色贴近 VS Code 主题，但整体对比和饱和度低于编辑器。
- 引用块使用轻微底色与左侧暖色标记区分，不做重型卡片。
- 脚注在桌面端以悬浮预览优先，避免跳转打断阅读；在移动端退化为行内展开或底部面板。

### 6.8 主题切换

- 页面提供一键夜间模式切换。
- 切换时，`PathBar`、`FileTree`、`ContentPane`、`ContextPanel` 统一切换到对应主题 token。
- 主题切换不改变页面结构与组件层级，只改变视觉变量。
- 主题偏好应跨会话保存。

### 6.9 空态交互

对未来文件类型的空态要统一：

- 显示文件类型名称。
- 说明该类型后续会支持的内容形式。
- 给出返回目录或打开其他可用文章的明确动作。

禁止出现：

- “404”
- “功能未实现”式硬错误语气
- 假装可用但点击后无反馈

## 7. 动效原则

动效只用于解释层级与状态，不用于装饰。

保留的动效：

- 文件夹展开/收起的轻过渡
- 文章切换时的轻量内容替换过渡
- 目录跳转时的平滑定位
- 当前章节同步高亮的细微变化
- 提示点展开文案时的轻微淡入

应取消或压弱的动效：

- 大面积浮动背景
- 过重的窗口开合动画
- 高频闪烁的选中态
- 与阅读无关的泛光、扫描线、终端打字效果

同时需要支持“减少动画”偏好，确保动效不会成为阅读障碍。

## 8. 响应式策略

### 8.1 桌面端

- 保留三栏。
- 中栏保持主位。
- 左右栏都可以固定，避免滚动中丢失定位。

### 8.2 平板端

- 收起右栏为抽屉或折叠区。
- 左栏仍然保留，但可适当缩窄。
- 正文优先级进一步提高。

### 8.3 手机端

- 退化为单栏正文。
- 左栏变为目录抽屉。
- 右栏目录合并为一个“本文目录”入口。
- 路径栏压缩为更短的层级表达。
- 脚注悬浮预览退化为点击后展开。

移动端不追求完整复刻桌面端文件系统感，优先保证：

- 能找到目录
- 能打开文章
- 能查看大纲
- 能返回上一级

- `DirectoryChildren`
- `DocumentView`
- `DocumentHeader`
- `ContextPanel`
- `TableOfContents`
- `FootnotePopover`
- `ProgressRestoreNotice`
- `FileTypeEmptyState`

每个组件的职责边界：

- `PathBar` 只做全局路径与位置提示。
- `FileTree` 只做浏览与定位。
- `ContentPane` 只做当前节点的主内容展示。
- `ContextPanel` 只做当前节点上下文。

不要在多个组件里重复承载同一类导航。

## 10. 验收标准

以下条件成立时，0.5 期设计可视为通过：

1. 用户首次进入后，3 秒内能理解页面结构与主任务。
2. 首次引导只帮助理解，不遮挡正文，不干扰后续操作。
3. 用户能明确知道当前打开的首页、目录或文件，以及它所在的路径位置。
4. 文件树在目录增多时仍可通过搜索和展开逻辑快速定位。
5. 首页节点能作为个性化入口存在，但不会把中间区做成另一套复杂首页系统。
6. 用户能在首页态、目录态和文件态之间自然切换，而不会把中间区误解为第二棵树。
7. 用户连续阅读长文时，不会被文件系统装饰感持续打断。
8. 用户能用右侧目录快速定位到文内章节，并在重新进入文章时恢复阅读位置。
9. 夜间模式切换后，各区域层级关系保持稳定。
10. 未支持的 `game` 和 `media` 文件不会造成误解或挫败。
11. 在窄屏环境中，目录浏览、文章阅读与脚注查看仍然可用。
12. 页面具备基础可访问性：足够对比度、清晰焦点态、键盘可达、动效可减弱。

## 11. 后续扩展方向

在不重做骨架的前提下，后续可扩展：

- `game` 文件的交互预览区
- `media` 文件的封面与播放器容器
- 文件夹级摘要页
- 按项目、标签、时间线切换不同视图
- 更完整的个人档案首页与内容入口

这些扩展都应复用当前“左侧目录 + 中央预览 + 右侧上下文”的基本逻辑。

## 12. 结论

这个页面的关键不是“做一个像 VS Code 的博客”，而是“做一个像个人文件系统的内容容器”。0.5 期先把文本阅读做好，把文件系统隐喻限制在浏览和定位层，再为未来更丰富的内容类型留出一致的结构空间。
